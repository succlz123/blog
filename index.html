Android R 文件的优化 / Shrink R Plugin

7月 9, 2021

https://github.com/succlz123/android-shrink
在 Android 体积优化中有个很有意思的点, 就是对 R 文件字段的内联, 这边记录下 R 文件的一些历史变更.

R 的变更

最早

http://tools.android.com/tips/non-constant-fields
当 App 与各个 lib 整合时，R 文件字段的实际值（必须是唯一的）可能会出现碰撞。ADT 14 之前，所有这样的字段都是 final 的，导致所有的 lib 被使用的时候, App Module 必须编译他们的所有资源和相关 Java 代码。

缺点

  • 构建速度非常慢
  • 阻碍了不包含源代码的 lib 项目的分发

优点

  • 字段不再是 final 的原因是意味着 jar 包可以只编译一次，并且能直接在其他项目中复用
  • 允许分发库项目的二进制版本（在 r15 出现）, 加快构建速度

ADT 14+ / 每个 lib 都带 R.java

从 app 里的 R.java 变到 每个 lib 都生成各自的 R.java , Module 里的 R 的变量 不是 static final 常量了

  1. 因为每个模块一个 r 文件,那么就会有冲突问题, 所以需要做 R 文件检查, 这样导致了必须编译他们的所有资源和相关 Java 代码, 所以分开
  2. 分开后, 如果还是常量的话, 那么冲突后再修改起来就太麻烦了, 直接用引用方式就好了

Android Gradle 3.3.0+ / 直接生成 R.class

https://developer.android.com/studio/releases/gradle-plugin?hl=ru

Faster R class generation for library projects: Previously, the Android Gradle plugin would generate an R.java file for each of your project’s dependencies and then compile those R classes alongside your app’s other classes. The plugin now generates a JAR containing your app’s compiled R class directly, without first building intermediate R.java classes. This optimization may significantly improve build performance for projects that include many library subprojects and dependencies, and improve the indexing speed in Android Studio.

Android Gradle 3.6.0+ / 减少重复的 R 文件生成

AGP 文档 / Simplified R class generation

The Android Gradle plugin simplifies the compile classpath by generating only one R class for each library module in your project and sharing those R classes with other module dependencies. This optimization should result in faster builds, but it requires that you keep the following in mind:

  • Because the compiler shares R classes with upstream module dependencies, it’s important that each module in your project uses a unique package name.
  • The visibility of a library’s R class to other project dependencies is determined by the configuration used to include the library as a dependency. For example, if Library A includes Library B as an ‘api’ dependency, Library A and other libraries that depend on Library A have access to Library B’s R class. However, other libraries might not have access to Library B’s R class If Library A uses the implementation dependency configuration. To learn more, read about dependency configurations.

https://github.com/objectbox/objectbox-java/issues/817

In 3.6 we started generating an R class per module and per AAR, and re-using them whenever possible (in libraries). That reduced the number of R classes from n^2 to linear. We still have to re-generate all R classes only at app level, since they need to use final resource IDs (in libraries they were non-final and mock values). To re-use these R classes we add them to the compile classpath, and this is why previously R classes would be visible to all consumers transitively, but now they’re only visible (at compile time) to consumers that have direct dependency on that module/AAR or the dependencies are “api” (transitive) and not “implementation”.

这里注意 3.6.0+ 修改后,   原来我们可能在别的模块里,也可能误调用到 别的 module 里的 R.id   现在这样之后就没这个问题了

编译时 AGP 对 R 文件的处理过程

  1. module 在 runtime 生成项目 class, 在 not_namespaced 里生成 R.class.jar
  2. 然后 R.class.jar 和 项目 class 一起被打包到 compile_library_classes_jar 里
  3. 然后 app 再根据 module 和依赖, 生成最终的 compile_and_runtime_not_namespaced_r_class_jar

为了重用优化

app 同时还有 compile_symbol_list

内联 R 文件

为什么不用 proguard 的优化

ProGuard 对 R 文件的混淆会发现也能删除很多 R$*.class，但是混淆会造成一个问题：混淆后不能通过反射来获取资源了。比如友盟统计分享等，就要求 R 文件不能混淆掉，否则会报错，我们常用的做法是开启混淆，但 keep 住 R 文件.

实现思路

Gradle 4.0 以后是从 jar 里读取了 - 不涉及到 directory 操作

  1. 从原来的 javac 下找 R.class   变成 从 app 的  compile_and_runtime_not_namespaced_r_class_jar/debug/R.jar 里找, 然后放入 map
  2. 遍历 jar -> 解压缩 -> 修改对应的 id -> 压缩回 jar (中间不涉及到  app 和 module 的 directory 的操作)

APK 中的 CERT.RSA 和 CERT.SF 文件丢失

10月 25, 2020

起因

前几天有应用需要发布上传市场, 然后在运营上传百度, 魅族, 小米市场时, 碰到说 apk 没有被签名. 但是这个问题很神奇, 为什么其他市场都正常呢, 而且 apk 可以正常的被安装到手机上去, 所以不可能没有签名.
解开 apk 文件后发现, 确实没有 *.RSA 的证书文件. 搜索半天发现也有人问过这个问题, 但是没人给出解答
https://android.stackexchange.com/questions/218187/why-cant-i-find-a-cert-rsa-file-in-my-android-apk
https://github.com/flutter/flutter/issues/58758

研究

  • 项目中的 build.gradle 看了半天也没发现有问题
  • 用 gradle 的 signingReport 命令也是能打出应用用的证书信息
  • 翻来覆去在 google developer 文档里面找, 也没有个头绪

https://developer.android.com/studio/command-line/apksigner
https://developer.android.com/studio/build/build-variants#signing
https://developer.android.com/studio/publish/app-signing?hl=zh-cn#release-mode
https://android-developers.googleblog.com/2016/11/understanding-apk-packaging-in-android-studio-2-2.html
https://developer.android.com/about/versions/nougat/android-7.0?hl=th#apk_signature_v2

解决

在我翻找资料时, 看到有如下写法, 所以尝试吧 v1 v2 都打开后, 打包出的 apk 里, 终于有 cert 文件了

为什么会没有 CERT 文件

在我加上 flag 解决没有 cert 文件问题后, 我思考半天想到是不是证书签名是否被打包到 apk 的二进制文件里面去了, 所以看不到.
同时我去 google 的 issuetracker 里找到了这个问题 https://issuetracker.google.com/issues/127290682
按照官方人员的回答, 我理解是打包签名出现问题是因为项目的 mini sdk version 大于了 23 , 在自动 build 的情况下, 直接忽略了 v1 签名. 而 v2 签名不需要 cert 文件, 当你用 keytool -printcert 去检查时, 当然会提示

除非用 Android sdk 下的 apk signature 工具检查

手动

手动通过 Android Studio 的 build 打包时, 我们通常会同时勾选 v1 和 v2 签名, 如果我们只勾选 v2, 那么在 apk 里其实也是找不到 cert 文件的

自动

目前有 v1SigningEnabled 和 v2SigningEnabled 2 个 flag
他们会产生四种结果：

  • v1 false & v2 false: 打包出来的 apk 就是未签名的应用，无法直接安装
  • v1 true & v2 false: apk 仅使用 v1 签名，兼容全部版本平台
  • v1 false & v2 true: apk 仅使用 v2 签名，应用仅可在 Android 版本大于等于 7.0 以上安装，在 7.0 以下无法安装
  • v1 true & v2 true: apk 同时使用 v1 v2 签名，兼容全部版本平台

当我们设置 mini sdk version 大于 23, 也就是最小版本大于 Android 7 时, 那么其实 gradle 自动判断不需要 v1 签名, 这是很合情合理的. 最后问题就出在国内市场还有只检查 v1 签名的存在.

Kotlin 使用小结

9月 21, 2020

没事总结下

优点

  • 空安全 -   内部实现 - 抛异常
  • 可以使用函数式编程 - 语法糖, 集合 高阶函数写的舒服 - apply let map - 实现逻辑 内部类 语法糖(lambda)
  • 集合中映射操作符
    map 转换 flatMap 铺平并转换
  • 扩展方法 -   内部实现 - 通过生成静态类和静态方法
  • 懒惰计算 lazy -   帮你生成一个, 你用到的是它重载了 get value，
  • 函数柯里化
  • 函数参数默认值
  • 尾递归 - 其实就是循环展开
  • inline 关键字, 自动内联提高性能
  • reified 关键字, 可以拿到泛型的 k class 信息,  kclass 就是 kotlin 对 java class 的一层封装 - 核心是编译阶段生成的 metadata 注解, 里面包括 class 很多信息
    • Kotlin 编译器会将 reified 方法asType内联(inline)到调用的地方(call-site)
    • 方法被内联到调用的地方后，泛型 T 会被替换成具体的类型

所以 reified 使得泛型的方法假装在运行时能够获取泛型的类信息 ,   而且 基于 inline 的  reified 是会          智能根据顺序 内联相应代码

  • ReadOnlyProperty - 一个接口 kotlin 的 delegate 和 observe 也都是基于这些做的 复写了 setValue getValue 方法 - Kproperty 的 getValue 功能 其实就是
    1. 比如 Kproperty1 的某个实现类 (在 class 中的 var 变量, 也就是带 receiver , 不可变就是 val 的变量引用) MutablePropertyReference1,
    2. 每当我们使用一次  class::xx 变量时, 会生成一个  helper class  xx$   继承自 MutablePropertyReference1
    3. 然后编译器 自动帮我们, 实现了他的各自接口, 没有反射, 本质就是 编译器自动生成   一个 帮助类, 只是一种 字节码生成功能
  • KClasses 扩展了许多跟反射相关的方法，是 kotlin 的反射中类主力输出。
    Java 是 class
  • 可见性
    1. 默认公开 public
    2. java 有 hide   kotlin 有 internal 包内可见
      编译时无法跨包调用，但是运行时 java 还是可以的，但是可以通过 @JvmName 或者 ``符号来生成 Java 不能调用的方法名字
    3. protected
      • Java 中 protected 表示包内可见 + 子类可见。
      • Kotlin 中 protected 表示 private + 子类可见。
    4. privated
      • Java 中，外部类可以访问内部类的 private 变量
      • Kotlin 中，外部类不可以访问内部类的 private 变量
  • receiver - 实现 DSL 功能
  • 密封类 - 子类类型的限定
    使用 when 表达式 的时候，能够验证语句覆盖了所有情况，就不需要为该语句再添加一个 else 子句了
    sealed class 在意 class 类型,  enum 在意 数据
  • 尾随逗号 - 1.4

缺点 or 不方便

  • sam 只能对 java 接口起效果，SAM 转换，即 Single Abstract Method Conversions，就是对于只有单个非默认抽象方法接口的转换。
    1. 官方的解释是 Kotlin 本身已经有了函数类型和高阶函数等支持，所以不需要了再去转换了。
    2. 只支持接口，不支持抽象类
    3. 某个版本之后已经可以支持 kt 的单接口
  • 默认 final 的 class 和 method，导致单元测试时的 mock 会有问题, 需要加注解 on Visible
  • 默认的空类型和非空类型，mock 时 也会有问题，mockito 返回的 any 对象是有可能是空的，需要专门引入 Mockito-Kotlin 的库。
  • koltin ktlint 插件
  • data class 没有默认的构造器，反射时需要
    使用 Kotlin 官方为我们提供了两个插件
  • 使用 reified 处理后的 kotlin 方法无法被 java 代码调用。
  • 扩展方法  java 和 kt 调用
    java 调用 泛型擦除后, 会冲突, 因为方法签名一样,   可以通过加  @JvmName 注解 来起 另外一个名字, 或者干脆换名字
    kt 里没关系, 因为编译器能区别
  • 自动生成 get set 与 fun getx 冲突
    @JvmName(“getxxx”) 起个别名
  • 各种注解
    JvmField   不要生成 getter/setter 并将其作为公开字段（public）
    JvmName 用注解名字替换
    JvmMultifileClass 生成一个多文件类，该文件具有在此文件中声明的顶级函数和属性作为其中的一部分 就是最后合并到一个 clas 里
    JvmOverloads   默认参数值的重载
    Strictfp 浮点数精度控制
    Synchronized Volatile Transient
    JvmDefault
    kt 的默认 interface 默认方法是通过生成 静态内部类 DefaultImpls 实现
    这样与 java 互操作就 看着有点奇怪, 加这个注解, 需要 1.8 +  Xjvm-default=enable 开启
    因为 java8 通过在 方法前加 default 可以支持 接口方法, 最后 kt 字节码 就是 java8 样子了
    还可以用  1.8 + -Xjvm-default=compatibility 开启兼容模式, 这样没加就是默认, 加了就是新的, 然后 没加的   静态方法就变成 调用 他自己的  default 方法
  • stb 自带 Pair 或 Triple
  • asType 快捷转换
  • reified - 具体化
    1. 在泛型类型前面增加 reified
    2. 在方法前面增加 inline（必需的）
  • @file 注解
    @file JvmName(“xxx”) - 如果 重名冲突 可以 JvmMultifileClass 合到一起

陷阱

  • getset 语法糖会默认生成很多 get set 方法，通常来说 kotlin 的代码会比 java 少很多。但是这个语法糖就有可能导致你的方法数量爆炸。
    可以通过 jvmfiled 注解避免。
  • 伴生对象的开销，非 const 的私有伴生对象，如果我们给他弄个 get set 方法，会生成很脏的代码，访问变量需要经过 4 个静态方法
    • JvmField 避免
    • 加 const
  • lazy()委托属性 就是帮你包装下
    lazy()委托属性可以用于只读属性的惰性加载，但是在使用lazy()时经常被忽视的地方就是有一个可选的 model 参数：lazy()默认情况下会指定LazyThreadSafetyMode.SYNCHRONIZED，不必要的性能开销。
  • 基本类型数组的拆装箱
    在 Kotlin 中有 3 种数组类型：
    • IntArray，FloatArray - 被编译成int[]，float[]
    • Array：非空对象数组 Array<Int> - 被编译 new Integer[]{}
    • Array：可空对象数组 Array<Int?> - 被编译 new Integer[]{}
  • Kotlin 提供了downTo、step、until、  reversed等函数来帮助开发者更简单的使用 for 循环，如果单一的使用这些函数确实是方便简洁又高效，但要是将其中两个结合呢？
    导致多个 IntProgression 对象的创建
  • 反射慢 , kotlin 比 java 多了很多反射信息, 所以特地弄了个 @metaData 所以慢很多, 还需要反射库 2.5M 混淆后 500KB

  2. kotlin 也可以有 asm
  3. 更智能的 类型推导

写的时候的注意

  • 使用 inline 修饰高阶函数（参数为函数时）
  • 把函数参数尽可能放到最后
  • 尽可能使用只读集合
  • 使用 lateinit 处理无法再构造函数初始化的变量
  • 善用 Data class 的 copy 方法

  • 使用含义更加清晰的 substringBefore 和 substringAfter
