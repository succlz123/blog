Kotlin 使用小结

9月 21, 2020

没事总结下

优点

  • 空安全 -   内部实现 - 抛异常
  • 可以使用函数式编程 - 语法糖, 集合 高阶函数写的舒服 - apply let map - 实现逻辑 内部类 语法糖(lambda)
  • 集合中映射操作符
    map 转换 flatMap 铺平并转换
  • 扩展方法 -   内部实现 - 通过生成静态类和静态方法
  • 懒惰计算 lazy -   帮你生成一个, 你用到的是它重载了 get value，
  • 函数柯里化
  • 函数参数默认值
  • 尾递归 - 其实就是循环展开
  • inline 关键字, 自动内联提高性能
  • reified 关键字, 可以拿到泛型的 k class 信息,  kclass 就是 kotlin 对 java class 的一层封装 - 核心是编译阶段生成的 metadata 注解, 里面包括 class 很多信息
    • Kotlin 编译器会将 reified 方法asType内联(inline)到调用的地方(call-site)
    • 方法被内联到调用的地方后，泛型 T 会被替换成具体的类型

所以 reified 使得泛型的方法假装在运行时能够获取泛型的类信息 ,   而且 基于 inline 的  reified 是会          智能根据顺序 内联相应代码

  • ReadOnlyProperty - 一个接口 kotlin 的 delegate 和 observe 也都是基于这些做的 复写了 setValue getValue 方法 - Kproperty 的 getValue 功能 其实就是
    1. 比如 Kproperty1 的某个实现类 (在 class 中的 var 变量, 也就是带 receiver , 不可变就是 val 的变量引用) MutablePropertyReference1,
    2. 每当我们使用一次  class::xx 变量时, 会生成一个  helper class  xx$   继承自 MutablePropertyReference1
    3. 然后编译器 自动帮我们, 实现了他的各自接口, 没有反射, 本质就是 编译器自动生成   一个 帮助类, 只是一种 字节码生成功能
  • KClasses 扩展了许多跟反射相关的方法，是 kotlin 的反射中类主力输出。
    Java 是 class
  • 可见性
    1. 默认公开 public
    2. java 有 hide   kotlin 有 internal 包内可见
      编译时无法跨包调用，但是运行时 java 还是可以的，但是可以通过 @JvmName 或者 ``符号来生成 Java 不能调用的方法名字
    3. protected
      • Java 中 protected 表示包内可见 + 子类可见。
      • Kotlin 中 protected 表示 private + 子类可见。
    4. privated
      • Java 中，外部类可以访问内部类的 private 变量
      • Kotlin 中，外部类不可以访问内部类的 private 变量
  • receiver - 实现 DSL 功能
  • 密封类 - 子类类型的限定
    使用 when 表达式 的时候，能够验证语句覆盖了所有情况，就不需要为该语句再添加一个 else 子句了
    sealed class 在意 class 类型,  enum 在意 数据
  • 尾随逗号 - 1.4

缺点 or 不方便

  • sam 只能对 java 接口起效果，SAM 转换，即 Single Abstract Method Conversions，就是对于只有单个非默认抽象方法接口的转换。
    1. 官方的解释是 Kotlin 本身已经有了函数类型和高阶函数等支持，所以不需要了再去转换了。
    2. 只支持接口，不支持抽象类
    3. 某个版本之后已经可以支持 kt 的单接口
  • 默认 final 的 class 和 method，导致单元测试时的 mock 会有问题, 需要加注解 on Visible
  • 默认的空类型和非空类型，mock 时 也会有问题，mockito 返回的 any 对象是有可能是空的，需要专门引入 Mockito-Kotlin 的库。
  • koltin ktlint 插件
  • data class 没有默认的构造器，反射时需要
    使用 Kotlin 官方为我们提供了两个插件
  • 使用 reified 处理后的 kotlin 方法无法被 java 代码调用。
  • 扩展方法  java 和 kt 调用
    java 调用 泛型擦除后, 会冲突, 因为方法签名一样,   可以通过加  @JvmName 注解 来起 另外一个名字, 或者干脆换名字
    kt 里没关系, 因为编译器能区别
  • 自动生成 get set 与 fun getx 冲突
    @JvmName(“getxxx”) 起个别名
  • 各种注解
    JvmField   不要生成 getter/setter 并将其作为公开字段（public）
    JvmName 用注解名字替换
    JvmMultifileClass 生成一个多文件类，该文件具有在此文件中声明的顶级函数和属性作为其中的一部分 就是最后合并到一个 clas 里
    JvmOverloads   默认参数值的重载
    Strictfp 浮点数精度控制
    Synchronized Volatile Transient
    JvmDefault
    kt 的默认 interface 默认方法是通过生成 静态内部类 DefaultImpls 实现
    这样与 java 互操作就 看着有点奇怪, 加这个注解, 需要 1.8 +  Xjvm-default=enable 开启
    因为 java8 通过在 方法前加 default 可以支持 接口方法, 最后 kt 字节码 就是 java8 样子了
    还可以用  1.8 + -Xjvm-default=compatibility 开启兼容模式, 这样没加就是默认, 加了就是新的, 然后 没加的   静态方法就变成 调用 他自己的  default 方法
  • stb 自带 Pair 或 Triple
  • asType 快捷转换
  • reified - 具体化
    1. 在泛型类型前面增加 reified
    2. 在方法前面增加 inline（必需的）
  • @file 注解
    @file JvmName(“xxx”) - 如果 重名冲突 可以 JvmMultifileClass 合到一起

陷阱

  • getset 语法糖会默认生成很多 get set 方法，通常来说 kotlin 的代码会比 java 少很多。但是这个语法糖就有可能导致你的方法数量爆炸。
    可以通过 jvmfiled 注解避免。
  • 伴生对象的开销，非 const 的私有伴生对象，如果我们给他弄个 get set 方法，会生成很脏的代码，访问变量需要经过 4 个静态方法
    • JvmField 避免
    • 加 const
  • lazy()委托属性 就是帮你包装下
    lazy()委托属性可以用于只读属性的惰性加载，但是在使用lazy()时经常被忽视的地方就是有一个可选的 model 参数：lazy()默认情况下会指定LazyThreadSafetyMode.SYNCHRONIZED，不必要的性能开销。
  • 基本类型数组的拆装箱
    在 Kotlin 中有 3 种数组类型：
    • IntArray，FloatArray - 被编译成int[]，float[]
    • Array：非空对象数组 Array<Int> - 被编译 new Integer[]{}
    • Array：可空对象数组 Array<Int?> - 被编译 new Integer[]{}
  • Kotlin 提供了downTo、step、until、  reversed等函数来帮助开发者更简单的使用 for 循环，如果单一的使用这些函数确实是方便简洁又高效，但要是将其中两个结合呢？
    导致多个 IntProgression 对象的创建
  • 反射慢 , kotlin 比 java 多了很多反射信息, 所以特地弄了个 @metaData 所以慢很多, 还需要反射库 2.5M 混淆后 500KB

1.40

  1. 参数列表最后的逗号
  2. kotlin 也可以有 asm
  3. 更智能的 类型推导

写的时候的注意

  • 使用 inline 修饰高阶函数（参数为函数时）
  • 把函数参数尽可能放到最后
  • 尽可能使用只读集合
  • 使用 lateinit 处理无法再构造函数初始化的变量
  • 善用 Data class 的 copy 方法
  • 针对函数类型和集合使用 typealias  A 里 尔 s
  • 使用含义更加清晰的 substringBefore 和 substringAfter
  • 巧用 Kotlin 反射实现按值取名，调试做到多快好省
